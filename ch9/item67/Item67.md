# Item 67. 최적화는 신중히 하라

- 빠른 프로그램보다는 좋은 프로그램을 작성해야 함
    - 좋은 프로그램은 각 요소를 독립적으로 설계할 수 있음(Item 15 참고)
    - 구현상의 문제는 나중에 최적화 하면 되지만, 아키텍처의 문제는 나중에 가서 해결하려고 하면 처음부터 다시 해야하는 경우도 생김

- 성능을 제한하는 설계를 피해야 함.
    - API, 프로토콜, 데이터 포맷 등의 설계 요소들은 완성 후에는 클라이언트가 이미 사용하고 있기 때문에, 변경하는 것이 어려움.
    
- 성능을 제한하는 설계의 예시
    - Item 50, 18, 64 참고

- 성능을 위해 API를 왜곡하는 것은 매우 좋지 않음.
    - 성능 문제는 플랫폼이 발전하면 사라질 수 있지만, 바람직하지 않은 설계의 API는 클라이언트들이 사용하고 있기 때문에 변경이 어려움.

- 프로파일링 도구 등을 이용해 정말 최적화를 해야할 곳만 하고, 전후 성능을 측정해야 함.
